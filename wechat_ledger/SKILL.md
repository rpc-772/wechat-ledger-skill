# Wechat Ledger

按下面流程处理一句话记账。

## 1. 触发条件

当用户发送短中文文本，且内容明显像记账句式时，直接进入本技能。

示例：

- `午饭32微信`
- `打车18支付宝`
- `咖啡26`
- `工资+12000`
- `报销300`

## 2. 抽取字段

产出以下字段：

- `Name`
- `金额`
- `类型`
- `分类`
- `支付方式`
- `时间`
- `备注`
- `原始消息`
- `已确认`

规则如下：

- 金额必须明确；若不能明确解析，只追问一次
- 类型默认 `支出`
- 出现 `+`、`收入`、`工资`、`奖金`、`报销`、`退款` 时，类型设为 `收入`
- 分类固定映射：
  - `午饭` / `早餐` / `晚饭` / `咖啡` / `奶茶` -> `餐饮`
  - `地铁` / `公交` / `打车` / `停车` -> `交通`
  - `工资` / `奖金` / `报销` / `退款` -> `工资`
  - 其他 -> `其他`
- 支付方式只识别：`微信`、`支付宝`、`银行卡`、`现金`
- 未写支付方式则留空
- 未写时间则使用当前本地时间
- 必须保留全文到 `原始消息`

## 3. 字段归一化

- `工资+12000` -> `Name=工资`
- `报销300` -> `Name=报销`
- `午饭32微信` -> `Name=午饭`
- `打车18支付宝` -> `Name=打车`

## 4. 写入 Notion

优先复用项目中已有的 Notion 能力；如果没有，再按这里的最小约定写入。

当前优先目标是现成模板中的 `鬆散的開支記錄 DB`。

环境变量：

- `NOTION_API_KEY`
- `NOTION_DATA_SOURCE_ID`

注意：

- 环境变量名保留 `NOTION_DATA_SOURCE_ID`
- 对 `鬆散的開支記錄 DB` 来说，这里实际填写的是它的 `database_id`

### 优先映射：`鬆散的開支記錄 DB`

- `Name` -> 标题列
- `金额` -> `金額`
- `时间` -> `購買日期`
- `分类` -> `類別`
- `备注` -> `備忘`

建议补充：

- `類型`
- `支付方式`
- `原始消息`
- `已確認`

如果这些字段不存在，可以只写入已有字段，但必须明确说明能力受限，不能假装完整成功。

### 备用映射：最小模板

- `Name`：Title
- `金额`：Number
- `类型`：Select
- `分类`：Select
- `支付方式`：Select
- `时间`：Date
- `备注`：Rich text
- `原始消息`：Rich text
- `已确认`：Checkbox

写入前检查：

- 缺少 `NOTION_API_KEY` 或 `NOTION_DATA_SOURCE_ID` 时，明确返回失败
- 没有可用的 Notion 写入能力时，明确返回失败
- 字段不兼容时，明确返回失败

若使用 `鬆散的開支記錄 DB`，优先按其现有字段名写入，并按 `database_id` 方式创建页面。

### OpenClaw 最小调用入口

如果当前 workspace 没有现成的 OpenClaw 路由或工具封装，直接调用本技能自带入口：

- 预览解析结果，不写入：
  - `python skills/wechat_ledger/scripts/run.py --message "{原始消息}"`
- 真正写入 Notion：
  - `python skills/wechat_ledger/scripts/run.py --message "{原始消息}" --write`

执行约定：

- 收到短中文记账句式后，优先走这个脚本入口
- 若脚本返回成功，只向用户回复一行确认
- 若脚本返回失败，直接转述失败原因，不要改写成成功
- 若脚本返回 `记账失败：金额不明确`，只追问一次金额

## 5. 回复规范

成功写入后只回复一行：

`已记：{Name} / {金额} / {支付方式或未填}`

失败时用一句短话明确说明失败原因，例如：

- `记账失败：未配置 NOTION_API_KEY`
- `记账失败：未配置 NOTION_DATA_SOURCE_ID`
- `记账失败：当前环境没有可用的 Notion 写入能力`
- `记账失败：Notion 字段与当前记账映射不兼容`
